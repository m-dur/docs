Plate 03 · component catalog

The moving parts, indexed

A conceptual map of the platform. Specifics intentionally left to the code.

i.

Backend

  • API blueprints grouped by domain — banking, brokerage, investments, analytics, net worth, rules, admin
  • Integration services wrap each provider: tokens, pagination, retries, idempotent upserts
  • ETL handlers normalize, dedupe, and apply user-defined categorization rules
ii.

Orchestration

  • Banking pipeline — several runs daily, snapshot + digest
  • Portfolio pipeline — weekdays after close, prices + brokers
  • Trigger rules favor availability: stale-but-present beats gaps
iii.

Data

  • Canonical transactions and balances across every institution
  • Per-broker positions, activities, and daily net-worth snapshots going back years
  • Pre-aggregated views keep heavy reporting queries fast
iv.

Clients

  • Web dashboard — multi-page desktop app with deep analytics
  • Mobile PWA — installable, tab-based, endpoints pre-shaped for small screens
  • Gestures layered where they matter: pull-to-refresh, swipe, long-press
v.

Notifications

  • Daily activity summaries and net-worth deltas via chat
  • Operational alerts — token expiring, pipeline stale, scheduler down
  • Everything lands in one place, ranked and readable
vi.

Infrastructure

  • Containerized and self-hosted, private-network binding
  • Public ingress mediated by a tunnel + identity provider — no open ports
  • Remote access layered behind a VPN overlay
Plate 04 · capabilities in detail

What the machine can do

01

Continuous net-worth history

Daily snapshots back-fill from statement-era records and continue forward with live API data — one unbroken line, years deep. In-transit amounts between accounts are tracked so mid-transfer snapshots never show phantom dips.

statement bridgingtransfer tracking
02

Lot-aware investment tracking

Per-symbol performance with realized and unrealized gains, lot-level cost basis, automatic split handling detected from market data, and multi-broker aggregation so one position stays whole across whichever providers hold it.

tax lotssplit detectionmulti-broker
03

Cash flow, honestly counted

Transfers between your own accounts are filtered out of income and expense roll-ups — no double counting. Auto-categorization with user-defined rules, per-transaction overrides, and reimbursement linking back to the original expense.

transfer exclusionrules enginereimbursements
04

Import without drift

Provider CSV exports ingest through the same idempotent path as the live sync — duplicate detection with price tolerance, identical categorization rules, zero import drift when history needs seeding.

bulk CSVidempotent writes
05

Quick answers, small screens

The PWA is built for the daily check-in: pre-aggregated endpoints make each tab load feel instant, covering summary, activity, cash flow, investments, accounts, income, payments, subscriptions, and net worth.

installableinstant tabs
06

Observed, end to end

A telemetry layer times every API route call and tracks what each page touches; a host-level probe checks pipeline liveness every 30 minutes from outside the orchestrator; and operational alerts — token expiry, stale data, scheduler down — land in chat before anything user-visible breaks.

route timingliveness probeops alerts

Security posture

the boring parts, done properly

✓Containerized services bound to private networks — nothing listens on the open internet
✓Public access mediated by a tunnel and identity provider, not port exposure
✓OIDC authentication with bearer tokens; no shared static keys in production
✓Provider credentials encrypted at rest, keys held outside the application
✓Remote access layered behind a VPN overlay — minimal ingress surface
✓Independent liveness monitoring that fails loudly, not silently
